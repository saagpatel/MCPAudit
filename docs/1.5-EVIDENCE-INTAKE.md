# MCPAudit 1.5 Evidence Intake

MCPAudit `1.4.3` is in a clean release state. The `1.5` line should harden
adoption and collect more fixture evidence before changing scoring semantics.

Tracked milestone: <https://github.com/saagpatel/MCPAudit/milestone/1>

## Work Items

- Add adoption smoke coverage for docs and examples:
  <https://github.com/saagpatel/MCPAudit/issues/66>
- Collect redacted config-health fixtures before new diagnostics:
  <https://github.com/saagpatel/MCPAudit/issues/67>
- Validate prompt/resource scoring before composite changes:
  <https://github.com/saagpatel/MCPAudit/issues/68>
- Decide `1.5.0` versus beta after evidence intake:
  <https://github.com/saagpatel/MCPAudit/issues/69>

## Guardrails

- Keep default adoption examples config-only unless the example is explicitly
  about reviewed connected scans.
- Keep fixture data redacted and local-only. Store credential key names, never
  credential values.
- Keep prompt/resource findings visible through `non_tool_risk` and policy
  gates. Do not merge them into `risk_score.composite` during this intake.
- Prefer tests that protect existing user-facing docs and examples over new
  scanner behavior.

## Done Criteria

- Docs and example workflows have smoke coverage.
- New config-health fixture cases are redacted and tied to current behavior.
- Any new scoring proposal is backed by prompt/resource fixture evidence.
- The release decision is explicit: ship `1.5.0` for adoption hardening or move
  to beta only when output-contract risk is low.
