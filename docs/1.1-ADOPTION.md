# MCPAudit 1.1 Adoption Notes

MCPAudit `1.1` adds `non_tool_risk`, an additive prompt/resource risk signal
for scans that find prompt/resource capability or injection findings. Existing
`risk_score.composite` behavior stays tool-centered.

## What Changed

- `risk_score.composite` still summarizes tool permission risk.
- `non_tool_risk` summarizes prompt/resource capability and injection risk.
- `non_tool_risk` may be `null` when no prompt/resource findings are present.
- Policy examples still gate on findings and pins, not on `non_tool_risk`.

## Reading JSON Output

Show each server's tool score and non-tool score:

```bash
jq '.audits[] | {
  server: .server.name,
  tool_risk: .risk_score.composite,
  non_tool_risk: (.non_tool_risk.composite // 0)
}' mcp-audit.json
```

List prompt/resource targets that produced capability findings:

```bash
jq '.audits[] | {
  server: .server.name,
  findings: [.capability_findings[]? | {
    target_type,
    target_name,
    category,
    severity
  }]
}' mcp-audit.json
```

List prompt/resource injection findings:

```bash
jq '.audits[] | {
  server: .server.name,
  injections: [.injection_findings[]? |
    select(.target_type == "prompt" or .target_type == "resource") |
    {
      target_type,
      target_name,
      severity,
      pattern_name
    }
  ]
}' mcp-audit.json
```

## Choosing A Policy

Start with the policy that matches how reviewed your MCP server set is:

| Policy | Use when |
|--------|----------|
| `local-review.yaml` | You are exploring a workstation and want a light first pass. |
| `balanced-team-ci.yaml` | A team wants CI signal without blocking all medium tool findings. |
| `reviewed-local-workstation.yaml` | A developer machine has reviewed servers and pin baselines. |
| `approved-servers-ci.yaml` | CI should allow only named reviewed servers. |
| `ci-strict.yaml` | CI should fail on medium-or-higher findings and drift. |
| `browser-automation-ci.yaml` | Browser automation servers are reviewed and expected. |

Example:

```bash
mcp-audit scan \
  --inject-check \
  --pin-check \
  --json mcp-audit.json \
  --policy examples/policies/balanced-team-ci.yaml
```

Exit code `2` means the report was written but the policy gate failed.

## Output Consumers

Use `examples/schemas/audit-report.schema.json` as the current JSON Schema for
the serialized report model. Consumers should still ignore unknown fields,
because compatible `1.x` releases may add optional fields.

Copy-paste consumer examples live in `examples/consumers/`:

- `parse_report.py` for Python pipelines;
- `parse-report.mjs` for Node.js pipelines.

For compatibility-sensitive integrations:

- parse `risk_score.composite` as the stable tool-centered score;
- parse `non_tool_risk.composite` only when `non_tool_risk` is not `null`;
- key finding routing on `target_type` and `target_name`;
- key SARIF routing on stable MCP rule IDs, not display text.
